# Bench results — knievel v0.1

First measured benchmark run for the v0 series. Pairs the criterion
micro-benchmarks in `benches/` (hot inner loops) with a sketch of the
macro-load harness in `bench/macro/` (HTTP-level decision throughput
against a running knievel + Postgres). The macro numbers are
**operator-run, not yet captured here** — see § 3.

The contract for this file lives in `TESTING.md` § 8 and
`REQUIREMENTS.md` § 9.2.

## 1. Methodology

- **Knievel revision:** `91e546a` (`v0.1.6` series).
- **Toolchain:** rustc per `rust-toolchain.toml` (1.86), `criterion`
  0.5.
- **Hardware:** GitHub Actions `ubuntu-latest` runner, 2 vCPU,
  7 GiB RAM. Linux 6.x. **Cloud runner timing is noisy by ±20%**;
  treat absolute numbers as ballpark, regression deltas as
  authoritative.
- **Criterion mode:** `--quick` for the v0 capture (single-shot
  measurements; full statistical sampling is reserved for nightly
  trend tracking — see § 4).
- **Knievel build profile:** `bench` (criterion default; `release`
  optimizations + debug assertions off).

## 2. Microbenchmark results

### `selection_pick` — `filter` → `priority` → `weighted_random` for one decision

Synthetic snapshot: N flights, 1 ad each, every flight matching the
placement (worst case for `filter` — every row makes it past
targeting). Real production snapshots see filter cut the candidate
set by ~10–100× before priority/weighted-random run; treat these
numbers as upper bounds.

| Flights (N) | per-decision time | throughput |
|---:|---:|---:|
| 10 | 555 ns | 1.8 M decisions/s |
| 100 | 14.0 µs | 71 K decisions/s |
| 1 000 | 1.00 ms | 1.0 K decisions/s |
| 10 000 | 99 ms | 10 decisions/s |
| 100 000 | 10.4 s | 0.1 decisions/s |

Observation: linear in N. Acceptable at the v0 scale (the seed
fixtures put a few flights per project; real production snapshots
look more like 100–1 000 eligible flights after targeting). The
10k+ numbers exceed the p50 ≤ 8 ms SLO target — this matters when a
single project ever exceeds ~1k post-filter eligible flights.

**Open issue (post-v0):** `selection::filter` rebuilds the
candidate vector per decision. A snapshot-side index keyed on
`(site_id, ad_type)` would cut filter work to O(matches), pushing
the breakpoint past 100k flights. Tracked as a Phase 6 candidate
(not yet filed; add when a real workload pushes us past the
breakpoint).

### `hmac_verify` — `/e/i/{signed}` and `/e/c/{signed}` per-request cost

Single-event verifier exercised on every event endpoint hit. Two
cases: (a) signature checks against the current secret, (b)
signature checks against the current secret then falls back to the
previous secret (rotation overlap).

| Path | per-request time |
|---:|---:|
| Hot (current secret matches) | 2.01 µs |
| Cold (falls through to previous secret) | 4.00 µs |

Observation: dominated by HMAC-SHA256 over a fixed-size record
(~50 bytes) plus base64url-decode. The cold path doubles the cost
because both `mac_matches` calls execute on a mismatch. The 8-hour
rotation overlap window makes the cold path a small fraction of
total event traffic; even sustained at 50% cold mix the budget is
~3 µs/event, far below the limiting cost (Postgres `COPY` batching).

## 3. Macro load benchmark — sketch

The macro bench is a `vegeta`/`oha` script driving `POST
/v1/projects/{p}/decisions` against a knievel binary built in
`--release` mode + a Postgres-tier-equivalent backing store, with a
synthetic project of 100k active flights drawn from a realistic
distribution.

Harness lives at `bench/macro/loadgen.sh` with the methodology + the
fixture-construction recipe documented inline. **No numbers
captured yet for v0.1** — runs require infrastructure outside the
v0 CI envelope (~db.r6g.large Aurora, dedicated traffic generator
node). First run lands as part of `v0.1.0` cut and gets backfilled
into this file before the tag.

Until that lands, the SLO targets in `REQUIREMENTS.md` § 9 stay
flagged **TARGET (unverified)**.

## 4. Regression policy

Per `TESTING.md` § 8:

- **Macro bench:** > 20% regression on any of (p50, p99, sustained
  QPS) blocks the release tag without an explicit waiver in the
  release-tagging PR.
- **Microbenchmarks:** > 30% regression opens an issue
  automatically (nightly trigger; not release-blocking).

Both policies measure against the previous release's entry in this
file. v0.1 is the baseline; v0.2 will be the first comparison
target.

## 5. Reproducing locally

```sh
# Microbenchmarks (~minutes for a full run; --quick is single-shot):
cargo bench --bench selection_pick
cargo bench --bench hmac_verify

# Macro (operator-run; needs a knievel + Postgres + load generator):
bash bench/macro/loadgen.sh
```

Numbers from non-CI hardware will differ; report any > 5×
discrepancy as a likely environment issue (CPU governor, container
limits, etc.) before assuming a regression.
