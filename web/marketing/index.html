knievel

Fearlessly fast ad delivery that steals the show.

Self-hosted, multi-tenant ad serving for publishers who want direct delivery without running an exchange. A single Rust binary, a generated Ruby gem, a Helm chart — all driven by one OpenAPI spec.

Quickstart API reference GitHub →

Why knievel

  • vs. Kevel. Domain model openly inspired by Kevel — advertiser → campaign → flight → ad, plus site / zone / priority / ad-type. Self-hostable on your own Postgres. No third-party data path.
  • vs. building from scratch. The boring parts ship: opaque-token auth, project-scoped row-level security, partition manager + leader election, snapshot loader, idempotency cache, :batchUpsert per resource, cursor pagination, and a generated Ruby gem with hand-written Enumerable wrappers.
  • vs. a generic ad framework. Opinionated about Postgres + OpenAPI 3.0 + multi-tenant. No pluggable storage backend, no GraphQL second-API, no per-tenant code. The narrowness is the point.
  • vs. an OpenRTB stack. Knievel is publisher-side direct ad serving. No bid-request flow, no SSP integration, no real-time auction.

Architecture in one diagram

┌──────────────┐     POST /v1/projects/{p}/decisions    ┌──────────────┐
│ Calling app  │ ──────────────────────────────────────▶│   knievel    │
│ (gem / curl) │ ◀──────────────────────────────────────│ (Rust/poem)  │
└──────────────┘                                        └──────┬───────┘
       ▲                                          in-mem       │
       │                                          snapshot     ▼
       │                                                  ┌──────────────┐
       └─ impression/click pings ─────────────────────────▶│   Postgres   │
          GET /e/i/<sig>                                   │  (config +   │
          GET /e/c/<sig>                                   │  partitioned │
                                                           │   events)    │
                                                           └──────────────┘

Decisions touch RAM only — the configuration snapshot is keyed by (project_id, resource) and refreshed on Postgres LISTEN/NOTIFY with a 5-second poll backstop. Events are buffered in an in-process channel and COPY'd to partitioned tables every 1–2 seconds. The partition manager runs in-process under a Postgres advisory-lock leader election. No second datastore. Full picture in ARCHITECTURE.md.

Quickstart (5 minutes)

Brings up Postgres + knievel + a seed sidecar that creates one of each resource and drops a bearer at tmp/knievel-dev-token: 

git clone https://github.com/knievel-ads/knievel
cd knievel
docker compose -f examples/compose/compose.yaml up -d
curl -fsS http://localhost:8080/healthz
# => ok

From Ruby, the Enumerable wrapper hides the cursor walk:

require "knievel/client"

client = Knievel::Client.new(
  host: "http://localhost:8080",
  access_token: ENV.fetch("TOKEN"),
)

client.advertisers(project_id).each      { |adv| puts adv.name }
client.advertisers(project_id).first(10)
client.advertisers(project_id).lazy.select(&:is_active).first(20)

Full quickstart with the deterministic project-id derivation, decision request, and teardown: README § Quickstart.

What's in v0

Decision API

POST /v1/projects/{project_id}/decisions with priority-tiered selection, weighted random, blocklists, force overrides (gated three ways). HMAC-signed /e/i/<sig> impression and /e/c/<sig> click endpoints with stable dedup_key across signing-secret rotation.

Management API

Full CRUD plus :batchUpsert for advertisers, campaigns, flights, ads, sites, zones — single Postgres transaction with per-row diagnostics. Cursor-paginated lists across the eight demand+inventory chains (?limit=N&cursor=…, default 50, max 500).

Events & reporting

Decision/impression/click rows buffered in-process, COPY'd to partitioned Postgres tables every 1–2 s. Hourly rollup table for cheap reporting reads. In-process partition manager pre-makes 4 days of partitions; drops past-retention.

Auth & tenancy

Opaque-token bearers (argon2id at rest) plus optional JWT via JWKS auto-discovery. Project-scoped Reader / Editor / Admin roles. Postgres FORCE ROW LEVEL SECURITY on every per-tenant table; CI-enforced cross-tenant suite covers every project-scoped endpoint.

Generated client

Ruby gem with idiomatic Enumerable walks. Spec-driven so a future Python or Go binding rides on the same OpenAPI surface. Local-dev default servers: baked in; consumers override host for production.

Observability

Structured JSON logs via tracing; OTLP-exported OpenTelemetry spans; Sentry error reporting. All three carry the same request_id / trace_id. /healthz, /readyz, /version, /openapi.json served live.

Deliberately not in v0: OpenRTB, experiment framework, reporting UI, second datastore. Per-row idempotency on POST creates moves to Phase 6. See REQUIREMENTS § 11.

Deploy

  • Helm: helm install knievel oci://ghcr.io/knievel-ads/charts/knievel
  • Container image: ghcr.io/knievel-ads/knievel:vX.Y.Z — multi-arch, cosign-signed, build-provenance-attested, tagged on semver tags only.
  • Compose: examples/compose/ for local + bring-your-own-Postgres.
  • Postgres: 14+ vanilla. Aurora, RDS, Cloud SQL, Supabase all supported. Only required extension is pgcrypto.

Status & stability

0.1.x is "surface stabilizing." The wire contract is forward-compatible from this point — every change is additive unless a 6-month deprecation window has elapsed. Generated client compatibility follows a major.minor mirror (gem X.Y.* ↔ server spec X.Y). Full contract in REQUIREMENTS § 6.4.
